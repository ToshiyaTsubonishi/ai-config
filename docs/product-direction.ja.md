# ai-config Product Direction

## この文書の位置づけ

この文書は、`ai-config` の**方針・あるべき姿・目指す姿**を固定するための north star です。

`docs/constitution.md` が「この repo が何を責務として持つか」を定義する文書だとすると、この文書は「その責務をどんなプロダクト価値につなげるか」を示します。

実装詳細、CLI の使い方、個別の運用手順はここでは固定しません。開発が煮詰まったときに、判断を戻すための基準だけを残します。

## 出発点

`ai-config` は、AI の実行エンジンそのものを作る repo ではありません。

目指しているのは、AI agent がその場で使うべき Skill / MCP を選び、必要なら計画を作り、安全な境界の向こうへ実行を渡せるようにする **control plane** です。

この前提は、外部の共通仕様で定めている次の思想とも整合させます。

- Container First
- Environment Agnostic
- Stateless by Default
- Config over Code
- Readable by Humans

つまり `ai-config` は、派手な runtime 機能を抱え込むよりも、**環境依存を増やさず、説明できて、運用に載せやすい構造**を優先します。

## 目指すプロダクト像

半年からその先にかけて目指す姿は、次の一文で表せます。

> 限定された実業務の中で、必要な Skill / MCP を安全に選び、失敗時は安全に止まり、利用実績を説明できる control plane

大事なのは「何でもできる AI 基盤」になることではありません。
大事なのは、特定の業務領域で「この基盤は実際に役に立ち、しかも運用に耐える」と言えることです。

## 北極星

判断に迷ったときは、次の順で優先します。

### 1. 選定品質

Skill / MCP の数を増やすことより、**自然言語の依頼に対して適切な候補を返せること**を優先します。

### 2. 実務価値の証明

広いユースケースを薄く触ることより、**限定された 1 つか 2 つの業務フローで明確な成功体験を作ること**を優先します。

### 3. 安全な失敗

完璧に自動化することより、**失敗時に嘘をつかず、安全に止まり、人に戻せること**を優先します。

### 4. 説明可能性

ブラックボックスに見える便利さより、**構造・境界・責任分界・運用前提を人間が説明できること**を優先します。

### 5. 環境非依存

ローカルでの一時的な便利さより、**コンテナ・設定注入・read-only runtime を前提とした再現性**を優先します。

## あるべき姿

`ai-config` が目指す「あるべき姿」は次です。

### Capability Broker である

あらゆる処理を自分で実行するのではなく、「何を使うべきか」を整理する起点であること。

### Planner を持つが Runtime を抱え込まない

複雑タスクでは plan artifact を作るが、実行 runtime の内部事情までは主責務にしないこと。

### Read-Only Surface を保つ

selector-serving は、build-time に整えた index を読むだけの公開面として保ち、実行時の可変要素を最小化すること。

### Provenance を崩さない

Skill / MCP の出所、所有者、import 経路を曖昧にせず、後から見直せる状態を保つこと。

### 人に引き継げる

将来、社内外のエンジニアが堅牢化や再実装を行う場合でも、何を残すべきか、何を分離すべきかが文書と構造から分かること。

## 非目標

次は魅力的に見えても、`ai-config` の主目的にはしません。

- 何でもできる万能 AI プラットフォームを目指すこと
- 実証前から multi-cloud / multi-LLM 抽象化を作り込むこと
- runtime orchestration の便利機能を repo の中心に据えること
- ホスト固有のセットアップや属人的な運用手順に依存すること
- デモ映えのために運用説明性を犠牲にすること

## 実務利用を見据えた到達目標

実務利用を見据える場合、最初の成功は「全面展開」ではなく、**限定領域での価値証明**です。

そのために `ai-config` が目指す到達点は次です。

- 特定の業務領域で使うべき Skill / MCP を安定して返せる
- 失敗時の fallback や手動導線を設計できている
- Open WebUI など既存 UI から接続して使える
- 利用履歴、検索結果、失敗理由を後から確認できる
- 「どの業務で、何が、どれだけ良くなったか」を説明できる

## 今後 6 か月の優先順位

次の期間で優先すべきものは、機能の広がりよりも証明の深さです。

### 最優先

- 実クエリに対する選定品質の改善
- 限定業務での成功事例づくり
- 安全な失敗と手動 fallback の整理
- 利用実績と改善履歴の記録

### 優先するが、主役ではない

- Open WebUI / MCPO などの接続導線の磨き込み
- approved plan artifact の活用拡大
- dispatch runtime との境界強化

### 後回しにする

- 広すぎる業務領域への横展開
- 先回りした抽象化
- runtime convenience を理由にした責務の混在

## 判断に迷ったときの問い

実装や設計で迷ったら、次を順番に問います。

1. この変更は選定品質を上げるか
2. この変更は限定業務での価値証明に近づくか
3. この変更は安全な失敗と説明可能性を保てるか
4. この変更は環境依存や属人性を増やさないか
5. この変更は repo の責務境界を曖昧にしないか

1 つ目から 3 つ目に効かず、4 つ目か 5 つ目を悪化させるなら、原則としてやらないか外へ逃がします。

## この文書を更新するタイミング

次の場合にだけ更新します。

- プロダクトの主目的が変わるとき
- 実務利用に向けた優先順位が変わるとき
- 環境非依存・責務分離・安全な失敗の前提を見直すとき

日々の実装メモや一時的な計画で頻繁に更新する文書ではありません。
