# ai-config リポジトリ現状評価と今後の開発ロードマップ

本書は、`ai-config` リポジトリが掲げる理念（`product-direction.ja.md` や `constitution.md` 等）に対して、現在のコードベースや機能・責務がどの程度合致し、完成しているかを機能単位で俯瞰・評価したものです。

また、後半には今後どのような優先順位で開発を進めるべきか（RAGの改善等を含む）のロードマップをまとめています。

---

## 1. 全体評価：理念と実装の驚異的な一致

一言で言えば、**「非技術者がAIを活用して構築したとは思えないほど、ソフトウェアアーキテクチャの基本原則が徹底されている、非常に優れたリポジトリ」**です。

通常、目先の課題をAIに解決させていくと、コードが肥大化して「何でもできるが運用不可能なGod Tool（万能ツール）」になりがちです。しかしこのリポジトリは、「Stateless（状態を持たない）」「Container First」「Config over Code（コードより設定）」という原則がディレクトリ構造や依存関係にそのまま反映されています。AIエンジンそのものを作るのではなく、「統制されたControl Plane（コントロールプレーン）」を作るという初期設計が完璧に機能しています。

---

## 2. 機能別・責務別の個別評価と改善点

### ① Registry / Indexing層 (カタログ管理とインデックス構築)
*   **役割:** 外部のSkillやMCPサーバー、スクリプトなどのツールを収集し、高速に検索可能なインデックス（ToolRecord）として構築する。
*   **対象ファイル:** `src/ai_config/registry/*` (mcp_parser.py, skill_parser.py等), `build_index.py`
*   **完成度: 非常に高い (80%)**
*   **評価:**
    `vendor_skills.yaml` などを正本として扱い、ツール群を `ToolRecord` という統一フォーマットに正規化して取り込む処理が綺麗に分離されています。特に、`build_index.py` によってインデックス構築を**事前（Build-time）に行う仕組み**が確立されている点が秀逸です。これにより、Cloud Runなどのランタイム環境をリードオンリー（Read-Only）に保つことができています。
*   **改善点:**
    現状で十分に機能していますが、将来的に新しい外部ツールの形式に対応する際のパース処理の追加が、より簡単に（プラグイン的に）行えるようになるとさらに良くなります。

### ② Retriever層 (検索・選定・RAG)
*   **役割:** ユーザーの自然言語の依頼（クエリ）から、適切なToolRecord（Skill/MCP）を検索・ランキングして提示する。
*   **対象ファイル:** `src/ai_config/retriever/*` (hybrid_search.py, query_intent.py)
*   **完成度: 実用化への過渡期 (50%)**
*   **評価:**
    セマンティック検索（意味の検索）とBM25（キーワード検索）を `RRF (Reciprocal Rank Fusion)` で統合するという、**現代RAGのベストプラクティスな「骨組み」はすでに完成**しています。
*   **改善点（★本リポジトリ最大の伸びしろ）:**
    *   **日本語対応（必須）:** 現在、キーワード検索の分割処理が `text.lower().split()`（空白区切り）になっているため、日本語のクエリが正しく分割・ヒットしません。`SudachiPy` や `MeCab` などの日本語形態素解析器の導入が急務です。
    *   **Embeddingモデルの本格化:** テスト用のダミーモデル（Hash）から、多言語対応の軽量かつ強力なモデル（`multilingual-e5-large` や OpenAIの `text-embedding-3-small` など）へ移行・運用固定する必要があります。
    *   **ルールベースの意図解釈からの脱却:** `query_intent.py` で `if "バグ" in q:` のようなハードコードで意図（Intent）を判定しています。これを小規模なLLMを活用した「動的なルーター」に進化させることで、精度が飛躍的に向上します。

### ③ Planner / Orchestrator層 (計画策定)
*   **役割:** Retrieverで得た候補を用いて、タスクを実行するための「承認可能な計画（Approved Plan）」を自動策定する。
*   **対象ファイル:** `src/ai_config/orchestrator/*` (planner.py, graph.py, nodes.py 等)
*   **完成度: 高い (75%)**
*   **評価:**
    ここでも「実行」を行わず、「計画（Plan）」の生成に特化している点が素晴らしいです。グラフ構造を用いて、候補検索〜計画生成〜検証（Validation）〜再計画（Replan）のワークフローがステート（状態）として堅牢に管理されています。
*   **改善点:**
    計画生成の質は「Retrieverがどれだけ的確なツールを拾ってくるか」に強く依存します。Retriever層の精度向上に合わせて、Planner側のプロンプト（プロンプトテンプレート）へ渡す情報を微調整していくことになります。

### ④ Contracts / Execution Boundary層 (契約と実行境界)
*   **役割:** `ai-config` が生成した計画と、実際の実行エンジン（Dispatch）との間の「境界」と「契約」を定義する。
*   **対象ファイル:** `src/ai_config/contracts/*`, `src/ai_config/executor/plan_boundary.py`
*   **完成度: 極めて高い (90%)**
*   **評価:**
    `ApprovedPlan` 等のデータモデルが明確に定義され、実行エンジンへの橋渡しがサブプロセス（Subprocess）経由で疎結合に行われています。「Runtimeを抱え込まない」という `constitution.md` の宣言が、まさにこの実装によって具現化されており、アーキテクチャの白眉と言えます。
*   **改善点:**
    現在、互換性のために `src/ai_config/dispatch/` フォルダがリポジトリ内に残っています。ロードマップに沿って、これを別パッケージ（`ai-config-dispatch`）へと完全に外部化（Externalization）を完了させることが最終ステップです。

### ⑤ MCP Server / Serving層 (提供インターフェース)
*   **役割:** 外部のAIエージェント（Open WebUI等）が `ai-config` を利用するための標準APIを提供する。
*   **対象ファイル:** `src/ai_config/mcp_server/*`, `deploy/cloudrun/*`
*   **完成度: 非常に高い (85%)**
*   **評価:**
    MCP（Model Context Protocol）の仕様に準拠し、各種ツールを外部に公開できています。また、Cloud Runデプロイ用のDockerfileやYAMLマニフェストが既に整備されており、ステートレスかつセキュアな本番運用がすでに想定・実現されています。
*   **改善点:**
    ダウンストリームのMCPサーバーへのプロキシ通信時のエラーハンドリング（タイムアウト設定など）、実運用を見据えた堅牢化が今後のタスクになります。

## 3. 今後の開発ロードマップ（次に何をAIに指示すべきか）

現在のリポジトリは「土台・アーキテクチャ」が完璧に整っているため、これからは**「中身のアルゴリズム（選定品質）」を磨き込むフェーズ**です。

以下の順番で着手（AIに指示）することをお勧めします。

1. **RAGの日本語対応と精度改善（最優先）**
   * *指示の例:* 「`ai_config/retriever/hybrid_search.py` のキーワード検索（BM25）において、現在は空白区切りになっているため日本語クエリに対応できていない。`SudachiPy`（または `MeCab`）を使った日本語対応のトークナイザーを実装してほしい。」
2. **検索精度を測るEval（評価）基盤の構築**
   * *指示の例:* 「検索精度が向上したか定量的に測るために、テストクエリと期待されるTool IDのペアを定義した評価用スクリプト（Eval）を作ってほしい。CIで回せるようにしたい。」
3. **Intent Extractionの高度化**
   * *指示の例:* 「`query_intent.py` のハードコードによる意図解釈を辞めて、LLM（API）を用いた動的かつ柔軟な判定ロジックに置き換えたい。」
4. **Dispatchの完全分離**
   * *指示の例:* 「現在リポジトリ内にある `dispatch/` フォルダの機能を完全に外部化し、`ai-config` 側から依存を断ち切るクリーンアップを行ってほしい。」

このロードマップに沿って進めることで、「限定された業務領域で確実に価値を出す」という目標に最短で到達することができます。
